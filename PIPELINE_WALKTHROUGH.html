1. TL;DR — The Pipeline in 30 Seconds

The platform runs 3 automated pipelines that feed each other:

  1. Scrape — Playwright browser visits real city permit portals, extracts commercial permit data, stores it in our database
  2. Enrich — Contacts imported from Google Sheets are enriched via PeopleDataLabs API (job titles, LinkedIn, phone) and verified via NeverBounce
  3. Research — User asks a question, Claude Haiku preprocesses it, Claude Opus generates a full research report with citations

All three run on Bull queues backed by Redis, with automated cron schedules for scraping and manual triggers for enrichment.

2. Pipeline 1: Permit Scraping

Let's walk through exactly how we extract commercial permit data from the City of Frisco, TX.

1Source: The Real Website

Frisco Portal: https://www.friscotexas.gov/614/Reports
Plano Portal: https://www.plano.gov/197/Building-Inspection

These are public municipal websites where cities publish building permit records. They're not APIs — they're regular HTML pages with tables. Our scraper reads them like a human would, but automatically.

2Scraper Launches a Browser

The system uses Playwright (same engine behind Chrome DevTools) to:

  • Launch a headless Chromium browser
  • Navigate to the city portal URL
  • Spoof a real Chrome 120 user-agent string (anti-detection)
  • Wait 2–5 seconds between page loads (mimics human behavior)
// What our scraper actually does (simplified):
const browser = await chromium.launch({ headless: true });
const page = await browser.newPage();
await page.setExtraHTTPHeaders({
  'User-Agent': 'Mozilla/5.0 ... Chrome/120.0.0.0 ...'
});
await page.goto('https://www.friscotexas.gov/614/Reports');

3Find Permit Links on the Page

The scraper scans all <a> tags on the page, looking for links that contain keywords like permit, commercial, or report. It follows up to 5 of these links to find the actual data tables.

// Discovers permit report links automatically:
const links = await page.$$eval('a', anchors =>
  anchors.filter(a => 
    /permit|commercial|report/i.test(a.textContent + ' ' + a.href)
  )
);

4Extract Data from HTML Tables

On each linked page, the scraper finds <table> elements and reads every <tr> row. Each row's cells are mapped to fields:

Cell Position Mapped To Example Value
Column 1permit_numberCOM-2026-04821
Column 2project_nameStonebriar Centre Retail Expansion
Column 3address2601 Preston Rd, Frisco, TX
Column 4owner_nameBrookfield Properties
Column 5gc_nameAustin Commercial LP

Additional intelligence is extracted automatically:

  • Project Type — Keywords like "build-out", "ground-up", "renovation" are detected → tagged as Interior Build-Out, Ground-Up, etc.
  • Dollar Values — Regex extracts amounts like $2,450,000 from text
  • Dates — Regex extracts dates like 03/15/2026 and converts to ISO format

5Commercial Filter (Only Valuable Permits)

Not every permit matters. The system applies a commercial-only filter before storing anything:

✅ Included (Commercial) ❌ Excluded (Residential)
office, retail, restaurant, warehouse, industrial, mixed use, multifamily, hotel, apartment, medical, school, tenant build-out, shopping plaza, data center single family, deck, fence, pool, sprinkler, water heater, re-roof

Value threshold: Any permit with estimated value ≥ $500,000 is automatically classified as commercial, regardless of keywords.

6Store in Database

// Each permit is upserted (insert or update if exists):
INSERT INTO permit_leads (
  permit_number, city, state, project_name, address,
  project_type, estimated_value, application_date,
  owner_name, gc_name, architect_name,
  source_url, portal_name
) VALUES (...)
ON CONFLICT (permit_number, city)
DO UPDATE SET ...   -- updates existing record if permit already scraped

7Geocode the Address

After storing, a separate job uses the Google Maps Geocoding API to convert each address into map coordinates:

GET https://maps.googleapis.com/maps/api/geocode/json
    ?address=2601 Preston Rd, Frisco, TX
    &key=GOOGLE_MAPS_API_KEY

→ Response: { lat: 33.1157, lng: -96.8234, zip: "75034" }

This enables the map view on the Permits page — every permit appears as a pin on a real map.

8Automation Schedule

This entire pipeline runs automatically every day at 6:00 AM CST.

No human intervention needed. New permits appear in the dashboard by the time the team starts work each morning.

Logging: Every scrape run is recorded in scrape_logs — records found, inserted, skipped, duration, and any errors.

3. Pipeline 2: Contact Enrichment

Starting from a Google Sheet of names and companies, here's how we build a fully enriched, verified contact database.

1Import from Google Sheets

User pastes a Google Sheets URL into the platform. The system uses Google Sheets API v4 (OAuth-authenticated) to pull all rows.

Smart column mapping — The system recognizes common column name variations:

Our Field Recognized Column Names
full_namefull_name, name, contact_name, person
emailemail, email_address, e-mail
companycompany, company_name, organization
designationtitle, job_title, position, role
phonephone, phone_number, mobile
linkedin_urllinkedin_url, linkedin, linkedin_profile

A row is imported if it has at least a full_name, email, or company. Each contact is tagged with source: 'google_sheet'.

2Enrich via PeopleDataLabs

The PDL enrichment job queries each contact against PeopleDataLabs' database of 1.5B+ person records:

// API call for each contact:
GET https://api.peopledatalabs.com/v5/person/enrich
  ?email=john@acme.com
  &name=John Smith
  &company=Acme Corp

// Response fills in missing fields:
{
  "full_name": "John Michael Smith",
  "job_title": "VP of Construction",
  "job_company_name": "Acme Construction Corp",
  "work_email": "john.smith@acmecorp.com",
  "mobile_phone": "+1-214-555-0142",
  "linkedin_url": "linkedin.com/in/johnmsmith",
  "location_locality": "Dallas",
  "location_region": "Texas"
}

Non-destructive enrichment: Existing data is never overwritten. If the contact already has a phone number, the PDL phone number is ignored. Only blank fields get filled in.

Rate-limited: 200ms pause between API calls to respect PDL's rate limits.

3Verify Emails via NeverBounce

Every email address is verified in real-time against NeverBounce's SMTP validation network:

// API call:
GET https://api.neverbounce.com/v4/single/check
  ?key=NEVERBOUNCE_API_KEY
  &email=john.smith@acmecorp.com

// Response:
{ "result": 0 }  // 0=valid, 1=invalid, 2=disposable, 3=catchall, 4=unknown
API Result Status Set Meaning
0validEmail exists, inbox is active
1invalidEmail doesn't exist or bounces
2disposableTemporary/throwaway email service
3catchallDomain accepts all emails (can't verify)
4unknownCould not determine

4. Pipeline 3: AI Research Generation

A two-model AI pipeline that turns a plain-English question into a structured research report.

1User Asks a Question

Example query: "Find all commercial construction projects over $5M in the Dallas-Fort Worth market starting Q2 2026, identify the general contractors, and suggest outreach angles."

The user also selects a research mode:

  • Full Research Plan — Comprehensive analysis with recommendations
  • Sources Only — Just the data sources and where to find information
  • Validation Checklist — Steps to verify and cross-reference data
  • Outreach Strategy — Actionable outreach plan based on the data

2Stage 1: Claude Haiku (Fast Preprocessing)

Model: claude-haiku-4-5-20251001 (fast, cheap)

Haiku's job is to:

  • Extract the core intent from the user's query
  • Identify the market, company, and request type
  • Filter noise and normalize the input
  • Structure the context for the next stage

This takes ~1-2 seconds and costs fractions of a cent.

3Stage 2: Claude Opus (Deep Research)

Model: claude-opus-4-5 (most capable, highest quality)

Opus receives the preprocessed context and a mode-specific system prompt. It generates a full research report including:

  • Relevant data from our database context (permits, contacts)
  • Recommended data sources (PDL, permit portals, MLS, LinkedIn)
  • Structured analysis aligned with the selected mode
  • Actionable next steps

4Token Tracking & Cost Logging

Every research session is logged with full cost transparency:

{
  haiku_input_tokens: 342,
  haiku_output_tokens: 186,
  opus_input_tokens: 1420,
  opus_output_tokens: 2856,
  estimated_cost: "$0.08",

All sessions are saved and viewable in the Research History tab.

5. End-to-End: How All Three Connect

Here's a realistic scenario showing how the pipelines work together for a real business use case.

Scenario: "A new $8M retail project was just permitted in Frisco. Find the decision-makers and prepare outreach."

Step Pipeline What Happens Trigger
1 Permit Scraping 6 AM cron scrapes Frisco portal → finds new permit COM-2026-04821 for "Stonebriar Retail Expansion" at 2601 Preston Rd, $8.2M, Owner: Brookfield Properties, GC: Austin Commercial LP Automatic (daily cron)
2 Geocoding Google Maps API converts "2601 Preston Rd, Frisco, TX" → lat: 33.1157, lng: -96.8234, zip: 75034. Permit now appears on the dashboard map. Automatic (after scrape)
3 AI Research User clicks on the permit, asks: "Who are the key contacts at Brookfield Properties and Austin Commercial for this project?" → Opus generates a research report with recommended contacts, titles, and outreach angles. Manual (user-initiated)
4 Contact Search User searches PeopleDataLabs directly from the Contacts page: company="Austin Commercial LP" → finds 12 matching executives with titles, emails, and LinkedIn profiles. Manual (user-initiated)
5 Enrichment PDL enrichment job fills in missing phone numbers, LinkedIn URLs, and verified job titles for all 12 contacts. On-demand (one click)
6 Email Verification NeverBounce checks all 12 emails → 9 valid, 2 catchall, 1 invalid. Team knows exactly which emails are safe to use. On-demand (one click)
7 Outreach Ready Team has: verified emails, phone numbers, LinkedIn URLs, the permit context, and an AI-generated outreach strategy. Ready to reach out. —

Total time from permit discovery to outreach-ready: Under 5 minutes of user interaction. The permit was discovered automatically at 6 AM — by the time the team logs in, it's already in the system.

6. Extension Plan: Adding More Data Sources

The architecture is designed to be extended. Here's the concrete plan for scaling from 2 cities to many sources.

How it works: Each new city is a single file (e.g., scrapers/dallas.scraper.js) that extends the base scraper class. The base class handles browser management, rate limiting, and data storage. The city file only needs to define the URL and CSS selectors specific to that portal.

Why Adding Sources Is Fast

The platform isn't built around specific data sources — it's built around data pipelines. Every new source follows the same pattern:

  1. Ingest — Scraper, API call, or file upload → raw data
  2. Normalize — Map source columns to our standard schema
  3. Store — Upsert into Supabase (conflict-safe, no duplicates)
  4. Enrich — Run through PDL + NeverBounce (already built)
  5. Surface — Data appears in the existing dashboard views

Steps 3-5 are already built and reusable. A new source only needs steps 1-2.
